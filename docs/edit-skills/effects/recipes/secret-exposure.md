# secret-exposure

> 系: 隠す · 既定の型: blur
> 接地: 想定(blur機能仕様) + craft · 観測: 2026-07-12は0件

**このチャンネルの承認済み実データには blur が無い。以下は機能仕様と安全上の
原則から構成した接地なしの分類であり、worked example も合成例である。**

## 一行定義

APIキー・パスワード・個人情報・アクセストークンなど、公開してはいけない情報が
画面に映るため隠す。

## 判定シグナル

### 語彙(transcript)

- **補助**: 「APIキー」「シークレット」「パスワード」「トークン」「メール
  アドレス」など、秘匿情報を表示・入力する発話
- **補助(craft)**: 発話がなくても漏えいは起こるため、語彙の不在を none の根拠にしない

### 座標(frames --ocr の box / av の motion 領域)

- **決定的**: 秘匿文字列を含む実在OCR boxがあり、blurを置けるrectが確定している
- **補助(craft)**: 入力欄全体ではなく、知覚が確定した値の領域を使う。ただしLLMは
  rectを縮小・推測しない

### 画面(frames / OCR テキスト)

- **決定的**: OCRに秘密らしい値が現れる。例: `sk-` / `ghp_` で始まる長い文字列、
  JWTらしい3区切り文字列、password欄の平文、個人のメール・電話番号
- **決定的(craft)**: 秘密か公開用識別子か判断できず、公開後の回復ができない
  可能性が残る。**迷ったら安全側のblurへ倒す**

### 時間・格子(アンカーの形)

- **決定的**: 秘匿値が見える全アンカーを覆う。発話区間だけに短縮しない
- **補助**: 値が更新・移動した場合は新しい実在rectを持つアンカーとして扱う

## 既定の型

`blur`。安全最優先。LLM は座標・時刻を出力せず、知覚由来のアンカーを選ぶ。

## 反例(この型を当てない場合)

```text
ドキュメントに掲載された例示値 "sk-example-redacted" で、実際の認証には使えないと
確認済み。→ secret-exposureを当てない。ただし確認できなければblur側へ倒す。

「APIキー」というラベルだけが映り、値はマスク済み(••••••)。
→ 秘匿値そのものは露出していないのでblurを重ねない。
```

## 紛らわしい隣

- `tiny-target` — 小さく読みにくい秘密は見せる対象ではない。判読性に関係なく
  `secret-exposure` を優先する
- `already-legible` — よく読めることは none の根拠にならない。読める秘密ほど
  blur が必要
