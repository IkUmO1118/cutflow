# restatement

> 系: 切る · 既定の処置: cut
> 接地: 実データ(2026-07-12, 23件)

## 一行定義

同内容を後でうまく言い直しているときの、**収束前の版**。文が完成せずに終わる
「未完成の言いかけ」もここに含む(母艦の旧 `false-start` を統合済み。理由は
[README](../README.md) の G1)。

## 判定シグナル

### 語彙(transcript)

- **決定的**: 後続の候補テキストが、本候補の内容を**包含する・言い換えている**。
  実データでは `reason` 文字列自体にこの関係が刻まれる
  (`#27に収束する言い直し前半` = 本候補は候補 #27 に収束する)
- **補助**: 本候補が助詞・接続助詞で終わる(文が閉じていない。「〜して」
  「〜けど」等)。「じゃなくて」「いや」等の自己訂正語が候補内にある

### 時間・格子(候補の形)

- **補助**: 収束先の候補と**似た尺**(同じ内容を言っているので、極端に短い/
  長いことは少ない)。収束前の版が2つの候補に分裂している(息継ぎ・小さな
  無音を挟んで前半・後半に割れる)のも典型
- 決定的シグナルではない。尺だけでは `stumble`(フィラー単独)と区別できない

### 音(plan.perception.audio)

- **補助**: `plan.perception.audio: true` のとき、本候補と収束先候補の間に
  無音区間が注記される(言い直すための間)
- 代替(既定オフ時): 上記の語彙シグナル(包含関係)だけで判定できる。音の
  注記は無くても困らない

### 画面(plan.perception.ocr / frames)

- 通常は無関係(restatement は発話の言い直しであり、画面変化を伴わないことが
  多い)。画面が同時に変わっている場合は先に `demo-wait` / `dead-air` の
  判定を優先する(発話が言い直しでも、画面側に情報があるなら残す側の理由が
  勝つ)

## 既定の処置

`cut`。収束先(後で言い直された完成版)を残し、本候補を切る。

## 反例(この分類を当てない場合)

- **後続候補が本候補を包含しない場合**。単に似た話題が2回出てくるだけで、
  片方が「もう片方の完成版」ではないなら `restatement` ではなく通常の
  keep が2つ並んでいるだけ。無理に片方を切らない
- **収束先が存在しない「言いかけ」**。次の候補で言い直されず、そのまま
  話題が変わってしまった未完成の発話は、内容を検討しても救済できないので
  `restatement`(収束前の版)ではなく `tangent`(本題と無関係)や単純な
  `dead-air` 側の判断を先に検討する
- 実際の失敗ではないが: `#11に内容が含まれる言い直し前半`(`~/Movies/cutflow/2026-07-12`
  の `cutplan.json`、時刻 [57.52-58.18])は「候補の一部だけが言い直しに
  含まれる」パターンで、LLM は候補全体を cut/keep のどちらかにしか倒せない
  (G3)。人間が GUI でさらに細かく keep/cut を分割した反転が実際に記録
  されている——これは「AI の判断が誤り」ではなく「候補粒度の限界」であり、
  `restatement` 自体の反例ではない

## 紛らわしい隣

- `duplicate-tail` — 処置の対象が逆(`restatement` は**前半**を切る、
  `duplicate-tail` は**完成後に残った末尾の断片**を切る)
- `stumble` — 弁別子は内容の有無。`restatement` は言い直された**内容**が
  存在する(包含関係がある)。`stumble` はフィラーのみで内容そのものが無い
- `tail-clip` — `restatement` で前半を切るとき、切る境界が収束先の直前の
  語尾に食い込んでいないか確認する(境界の攻めすぎに注意)

## worked example

`~/Movies/cutflow/2026-07-12` の `cutplan.json` / `whisper-out.json` から。
「JSON バリデーションの効能」を説明する場面で、話者が一度息継ぎを挟んで
言い直している。

収束前の版(cut。2つの raw segment に分裂):

```text
[223.89-227.94] "壊れたJSONのまま数分かけてレンダリングして"
[227.94-228.44] (無音 0.5秒 — 言い直しの間)
[228.44-230.71] "初めて気づくみたいな事故を数秒で防げます"
```

収束先(keep。候補 #27。同じ内容を一息で言い切っている):

```text
[233.44-234.73][235.00-240.18] "壊れたJSONのまま数分かけてレンダリングして
初めて気づくみたいな事故を数秒で防げます"
```

実際の `cutplan.json` の `reason` は両区間とも `"#27に収束する言い直し前半"`。
以下の `id` は判断 JSON の形を示すための例示番号(この収録の `plan.raw.txt` は
最後の実行結果で上書きされており、生成当時の候補通し番号そのものは残って
いない。`reason` に残る `#27` という数字だけが実際の値)。

```json
{ "id": 26, "reasonId": "restatement", "reason": "#27に収束する言い直し前半" }
```
