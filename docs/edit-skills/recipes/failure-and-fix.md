# failure-and-fix

> 系: 残す · 既定の処置: keep
> 接地: 想定(実データ0件・`prompts/plan-cuts.md` 由来)

**この分類は `~/Movies/cutflow/2026-07-12` に実例が無い(想定由来)。**
その1本(「ツール紹介・デモ」)はほぼノーエラーで進行しており、失敗と
その解決をセットで見せる場面が発生していない。以下の判定シグナルと
worked example は `prompts/plan-cuts.md` の既存の判断基準(「失敗は
見せ場」)と一般的な編集知識から構成した**合成例**である(G4)。

## 一行定義

エラー・失敗と**その解決**(セットで残す。片方だけ残さない)。

## 判定シグナル

### 語彙(transcript)

- **決定的**: 「あれ」「失敗した」「エラーが出た」「動かない」等の発話が
  あり、**後続に解決(原因の発見・修正・再試行の成功)が続く**
- **補助**: 試行錯誤の過程を示す発話(「あ、これじゃない」「じゃあこっちを
  試してみる」)

### 時間・格子(候補の形)

- **決定的**: `failure-and-fix` は**単一候補ではなく範囲で判断する**。
  エラー発生の候補と、解決に至る候補までを1セットとして扱う(G3: LLM は
  候補を分割できないため、セット内の一部だけを cut することはできない。
  セット全体を残す判断をする)

### 音(plan.perception.audio)

- 決定的シグナルではない

### 画面(plan.perception.ocr / frames)

- **補助**: 画面 OCR にエラーメッセージ・赤字の警告・スタックトレース等が
  映っている場合、失敗の裏付けになる。解決後は該当のエラー表示が消えて
  いることが確認できる

## 既定の処置

`keep`。

## 反例(この分類を当てない場合)

- **失敗したまま解決に至らない場合**。エラーが出た後、話題が変わって
  そのまま流れてしまうなら、セットとして成立しないので `failure-and-fix`
  ではない(解決が無い失敗だけを残すと視聴者に不安だけが残る)
- **解決が「編集で無かったことにする」レベルの軽微なミス(言い間違い等)
  の場合**。これは失敗と解決のセットではなく、単なる言い直しなので
  `restatement` を検討する

## 紛らわしい隣

- `demo-wait` — 無言を挟む点で近いが、`demo-wait` は結果**待ち**、
  `failure-and-fix` は失敗**からの回復**。直前の発話がエラーに言及して
  いるかどうかで区別する
- `tangent` — エラー対応中の独語・脱線的な試行錯誤は一見脱線に見えるが、
  本題(ツールの動作)に直結しているので `tangent` ではなく
  `failure-and-fix` を優先する

## worked example

**接地なし。合成例。** コマンド実行がエラーになり、原因を特定して再実行
する、という想定シナリオ。

```text
[想定 300.0-303.5] "じゃあこれを実行してみます"
[想定 303.5-306.0] （コマンド実行。画面にエラーメッセージ）
[想定 306.0-309.2] "あ、エラーが出た。パスが間違ってるかも"
[想定 309.2-312.0] （修正して再実行。画面に成功メッセージ)
[想定 312.0-314.5] "はい、これで直りました"
```

このセット全体([300.0-314.5])を残す。途中のエラー画面([303.5-306.0])
だけを取り除くと、視聴者が「なぜ話者が慌てているのか」を追えなくなる。

```json
{ "id": 41, "reasonId": "failure-and-fix", "reason": "パス誤りのエラーと修正の一連" }
```
