# duplicate-tail

> 系: 切る · 既定の処置: cut
> 接地: 実データ(2026-07-12, 16件)

## 一行定義

完成版を言い切った**直後に残った**末尾・遷移句の反復。`restatement` とは
切る側が逆(`restatement` は**前半**を切るが、`duplicate-tail` は**後に
残った断片**を切る)。

## 判定シグナル

### 語彙(transcript)

- **決定的**: 本候補の先頭が、**直前の keep 候補の末尾**と一致・ほぼ一致する
  (単語の一部が千切れて繰り返される・同じ語を言い直そうとして出た断片)
- **補助**: 「というわけで」「はい」等の遷移句の反復。文として独立して
  成立しない断片(「〜ディ」「〜ター」のような単語の途中)

### 時間・格子(候補の形)

- **補助**: 尺が非常に短い(実データでは 0.2〜0.6s が大半)。直前の keep
  候補の直後に位置する(gap がほぼ無いか、短い無音を挟むのみ)
- 決定的シグナルではない。短尺だけでは `gap-trim` と区別できない
  ——`duplicate-tail` は候補内に**直前と同じ音・語の断片**があることが必要

### 音(plan.perception.audio)

- **補助**: 直前候補との間に無音がほぼ無い(言い終わった勢いでそのまま
  漏れ出た音)。`gap-trim` と違い、無音区間そのものは判定の主眼ではない
- 代替(既定オフ時): 語彙シグナル(直前末尾との一致)だけで判定できる

### 画面(plan.perception.ocr / frames)

- 通常は無関係

## 既定の処置

`cut`。

## 反例(この分類を当てない場合)

- **断片が直前候補の末尾と一致せず、次の文の助走(息を吸う音・言い出しの
  頭)である場合**。次に続く発話の一部なら切ると `reference-orphan` の
  リスクがある(次候補の内容が宙に浮かないか確認する)
- **断片に見えて、実は独立した新しい情報を含む場合**(例: 反復に見えて
  実際は言い直しで訂正が入っている)。その場合は `restatement` を検討する
- G3(候補の一部だけを切る操作は分類にしない): 実データの `#79末尾の重複`
  (`[546.93-547.13]` `[548.40-548.66]`)のように、末尾の重複が2つの短い
  candidate に分かれて別々に切られることがあるが、これは LLM が2つの候補を
  それぞれ `cut` と判断した結果であり、1つの候補を分割したわけではない
  (G3 の対象外)

## 紛らわしい隣

- `restatement` — 切る側が逆。`duplicate-tail` は完成後に残った末尾を切り、
  `restatement` は収束前の前半を切る
- `reference-orphan` — 末尾の断片を切った結果、直後の候補の指示語
  (「これ」「それ」)の参照先が本候補にしかない場合は `reference-orphan` の
  判定を優先する(末尾の断片自体は情報を持たないが、参照の踏み台になって
  いることがある)

## worked example

`~/Movies/cutflow/2026-07-12` の「GUI エディター」の話題転換部分。話者は
「GUIエディターも作りました」を言い切った直後、単語の一部を言いかけて
すぐ止め、0.65秒後に改めて言い直している。

```text
[356.35-358.35] (keep) "GUIエディターも作りましたN"  ※次候補の頭が混入
[360.31-360.82] "ディ"           reason: "#47末尾の重複"
[361.47-362.07] "ター"           reason: "#47末尾の重複(二重)"
[364.00-366.49] (keep) "動するとこう"  ※「npm run editorで起動するとこういう画面が…」の一部
```

「エディ…ター」は「エディター」という単語が千切れて2回に分かれて出た残響で、
直前の keep 候補「GUIエディターも作りました」の内容を1文字も追加しない。
