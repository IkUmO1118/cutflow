# demo-wait

> 系: 残す · 既定の処置: keep
> 接地: 実データ(反転keep4件, 2026-07-12) + rules.md

## 一行定義

**画面が語っている**無言(実行結果の待ち・出力を読ませている数秒・結果が
出る瞬間)。母艦の旧 `reveal` を統合済み(理由は [README](../README.md) の
G1)。

## 判定シグナル

### 語彙(transcript)

- **決定的**: 発話が無い
- **補助**: 直前の発話が「保存すれば」「実行すると」「これをやると」等、
  操作の宣言・結果を見せる予告になっている(=この後に無言で結果を見せる
  流れが自然に続く)

### 時間・格子(候補の形)

- **補助**: 実データでは 0.3〜0.5秒程度の短い無言。ただし `demo-wait` は
  尺の帯で `dead-air` と区別できない(長い結果待ちもありうる)。**尺は
  弁別子にしない**

### 音(plan.perception.audio)

- 決定的シグナルではない。`dead-air` と同じ無音注記を受ける

### 画面(plan.perception.ocr / frames)

- **決定的**: 画面 OCR / frames が**直前候補から変化している**
  (=保存・実行・操作の結果が画面に反映されている)。または直前発話が
  コマンド実行・操作の宣言になっている
- **この決定的シグナルが取れないとき(既定オフ環境)は、`dead-air` へ
  倒さず demo-wait 側(残す)へ倒す。** `rules.md` の失敗記録(下記反例)は
  逆方向(demo-wait を dead-air と誤判定して切った)で起きているため、
  判断に迷う場面では「切って良いのは画面にも新しい情報が無い段取りの間
  だけ」という `rules.md` の基準を優先する

## 既定の処置

`keep`。

## 反例(この分類を当てない場合)

- **画面が直前から変化していない場合**。話者が結果を見せる意図の発話を
  していても、実際に画面が動いていない・変化が確認できないなら
  `demo-wait` ではなく `dead-air`(段取りの間)を検討する
- **実際に起きた失敗(`rules.md`、2026-07-17)**: AI がデモ・GUI 操作中の
  無言画面を「静止画面の待ち時間」として大量カットした。正しくは画面に
  結果が出ている・出ようとしている `demo-wait` だった。この失敗を受けて
  `rules.md` は「デモ・画面操作中の無言は『待ち時間』ではなく見せ場」
  「切ってよい無言は画面にも新しい情報がない段取りの間だけ」と明記して
  いる
  → `~/Movies/cutflow/2026-07-12` の最終 `cutplan.json` には、この失敗が
  修正された痕跡が `action: "keep"` かつ `reason: "無音"`(AI が cut と
  判断した理由の文字列がそのまま残った反転)として4箇所残る:
  `[448.29-448.59]` `[457.78-458.30]` `[465.46-465.91]` `[468.45-468.87]`
  (下記 worked example)。いずれも GUI エディターの機能説明の直後——
  「保存すればホットリロードでその場に反映されます」「最終確認や細かい
  微調整はだいたいこっちでやってます」「作るときにいくつか意識したポイント
  があります」「一番気にしたのは料金です」——という**結果や次の話題を
  見せる予告**の直後に位置する

## 紛らわしい隣

- `dead-air` — 唯一の弁別子は画面の変化の有無。判断に迷ったら
  `demo-wait`(残す)側に倒す
- `failure-and-fix` — 結果待ちの無言(`demo-wait`)か、エラーからの回復
  待ち(`failure-and-fix`)かを区別する。直前の発話がエラー・失敗に言及
  しているなら `failure-and-fix` を優先する
