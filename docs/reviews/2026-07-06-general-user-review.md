### 1-1. ドキュメントとクイックスタート … ◎(一点だけ穴)

**強み**
- `docs/getting-started.md` が「このページだけで一通り動く」単一ページ導線。対象者別
  リンク(CLI / GUI / AI)・パイプライン図・ファイル責務表・コマンド表・
  トラブルシュート(§8)まで網羅。ソロ OSS の onboarding として水準が高い。
- README / getting-started / usage / recording-guide の4層で粒度が分かれ、重複が少ない。
- 「時刻は元収録の秒」「run は1回だけ」等の**落とし穴を先回りで明記**している。

**穴(要改善 → D1)**
- 既定の `llm.backend: claude-cli` は **`claude` CLI(Claude Code)のインストール+
  認証が前提**なのに、セットアップ手順(README §セットアップ / getting-started §1)は
  `brew install ffmpeg whisper-cpp` と `npm install` しか挙げていない。既定設定のまま
  `run` すると `plan` 段で `コマンド 'claude' が見つかりません`(`src/lib/exec.ts:35`)に
  当たる。**一般ユーザーが最初に必ず踏む段差**。§8 トラブルシュートにも該当項目が無い。
- Node 23.6+ という非常に新しい要件・macOS 専用(`open`/videotoolbox/brew 依存)は
  記載あり。CONTRIBUTING/CHANGELOG は無い(ソロ運用なので軽微)。

### 1-2. 機能の説明と操作方法(画像) … △(最弱ディメンション)

- `docs/` 配下に**画像ファイル0枚、Markdown 内の画像参照0件**(`find`/`grep '!['` で確認)。
- テキストと ASCII パイプライン図・表で相当量を補ってはいるが、以下は**本質的に図が要る**:
  - **拡張キャンバス方式**(3840×1080 左=画面/右=カメラ)の空間配置
  - **GUI エディタの画面構成**(左タブ / 右プレビュー / 下タイムライン / ヘッダー)
  - **ワイプの位置・大きさ**、テロップ座標系、素材の rect 配置
- CLI 対話(Claude Code で「◯◯を編集して」)の実例スクショや、before/after も無い。
- 真のスクリーンショットは**実収録が必要**でメンテナが撮る領域。ただし空間配置と
  エディタ画面は **ASCII 構造図で相当程度カバー可能**(→ Phase 2 で緩和)。

### 1-3. セキュリティ / プライバシー … ○

**ローカルサーバ(`editor/server.ts`)の守りは堅牢**(単一ユーザーのローカルツールとして十分以上):
- `127.0.0.1` バインド(`:127`)。外部インターフェースに晒さない。
- **DNS rebinding 対策**: Host ヘッダをローカルのみ許可(`:176-180`)。
- **CSRF 対策**: 非 GET は Origin がローカルのときだけ通す(`:181-184`)。
- **パストラバーサル対策**: `/media`・`/api/material`・`/api/reveal`・`getPeaks` すべて
  `normalize` 後に収録フォルダ前方一致で弾く(`:283, :333, :426, :726`)。
- アップロードは拡張子ホワイトリスト+サイズ上限(既定2048MB、chunked もストリームで打切り `:570`)。
- `.env` は gitignore、API キーはログに出さない。`spawn("open", …)` は固定引数で注入余地なし。
- 外部送信テレメトリ無し。

**プライバシー上の注意(要改善 → D2)**
- `plan`/`remeta`/`plan-shorts` は **文字起こしテキスト(+あれば `brief.md`)を Anthropic へ送る**
  (`src/lib/llm.ts`: claude-cli サブプロセス or `api.anthropic.com`)。**映像・画面・音声・
  whisper 文字起こし処理自体はローカル完結**という良い設計だが、「**喋った内容(=transcript)は
  外部 LLM に渡る**」ことがユーザー向け文書のどこにも書かれていない。画面に機密が映る
  リスクは recording-guide が警告するが、発話内容の外部送信は未記載。機密を喋る収録が
  ありうる一般ユーザーには明示すべき。

**軽微(記録のみ、今回は実装しない → V2)**
- `readBody`(`:771`)は JSON ボディにサイズ上限が無い(ローカル専用ゆえ低優先)。
- localhost サーバに認証は無い(単一ユーザー・ローカル前提として許容範囲)。

### 1-4. AI による編集機能 … ◎(本ツールの白眉)

- **編集=収録フォルダの JSON**という単純明快なモデル。AI がコードを触らず成果物を編集。
- **AI の知覚手段が揃っている**: `transcript.json`(何をいつ喋ったか)/ `describe`(元秒⇔
  出力秒対応のタイムライン要約)/ `frames --t|--captions|--every`(最終合成と同じ見た目で
  PNG 出力→ Read で自己確認)。「編集→validate→自己確認」を CLI で完結できる。
- **多重ガードレール**: `plan`/`run` 再実行禁止・`--force` を AI 判断で付けない・`approved` を
  AI が反転しない(承認は人間)・中間生成物と backups/ は不可侵・`--force` 時は手編集を
  backups/ へ退避。`CLAUDE.md` にこれらが精緻に明文化されている。
- **限界も文書化済み**: frames はプロキシのアップスケールなので画面内細字の可読性は判断不可
  (最終判断は人間の preview/render)。この自己認識の正直さが良い。

### 1-5. 標準NLE(Premiere / Final Cut Pro)との比較 … ○(意図的な絞り込み)

**cutflow が持つ編集能力**: カット(無音検出+LLM 意味カット)/ 多トラック・テロップ
(位置・色・フォント・座布団)/ 素材オーバーレイ(全画面・PiP rect)/ インサート編集 /
ワイプ全画面+遷移 / ズーム(Ken Burns 的)/ ディップ・トゥ・ブラック / 簡易カラー調整
(明るさ・コントラスト・彩度)/ BGM(区間配置・発話ダッキング・フェード)/ 章立て /
サムネイル / ショート(縦・複数プロファイル)/ マイク+システム音声ミックス+
-14 LUFS 自動正規化 / ノイズ除去 / 波形表示・マグネット吸着・undo・ホットリロード。

**汎用NLE比で意図的に持たない機能**(欠落ではなく設計上の割り切り):
- 速度変更 / タイムリマップ(`next-features-design.md` で timelapse として据え置き)
- トランジション集(クロスディゾルブ等。現状は dip-to-black とワイプ遷移のみ)
- カラーグレーディング(カーブ/ホイール/LUT。現状は全編一律の3値のみ)
- マスク / クロマキー / 高度な合成(現状は矩形配置まで)
- クリップ単位の音量エンベロープ・オーディオエフェクト(EQ/コンプ/ゲート。
  現状はスカラー音量+フェード+全体ラウドネス+BGM ダッキング)
- モーション・キーフレーム(ズームのイーズは固定)/ タイトルアニメ / MOGRT
- マルチカム(2分割キャンバスを超える)/ ネストシーケンス / 調整レイヤー
- 書き出しプリセット・フォーマット選択(mp4 固定パイプライン)

→ これらの不在は**「OBS 画面収録+カメラ → YouTube」という単一ワークフローに特化した
オピニオネイテッドな道具**であることの帰結。汎用NLEの代替ではなく、その前段の
「一次編集を自動化して人が仕上げる」層。**問題は不在そのものではなく、「持たない」ことが
どこにも明示されず、汎用NLEを期待した一般ユーザーが誤解しうる点**(→ D3)。

### 1-6. サービスとしての総合評価 … ○(対象を選べば成立)

- **技術者個人クリエイター向けの OSS ツールとしては成立し、かつ完成度が高い**:
  差分キャッシュ(cut/render/proxy/chunk)・プロキシ飛び飛び再生・承認ゲート・
  ホットリロード・AI 自己検証ループまで、ソロ製とは思えない作り込み。
- **差別化軸が明確**: ローカルファースト(映像は外部に出ない)/ 文字起こし無料 /
  LLM は BYO(Claude Code サブスク or 従量 API)。クラウド編集 SaaS に対する
  プライバシー・コスト優位。
- **汎用コンシューマ製品としては成立しない**: ターミナル / Homebrew / Node 23.6+ /
  OBS 拡張キャンバス設定 / (既定では)Claude Code サブスク が前提。ここでの
  「一般ユーザー」は現実には**技術リテラシーのある個人**。この対象設定は妥当だが、
  1-1〜1-3・1-5 の3つの「明示されていない前提/範囲」を埋めれば、対象内ユーザーの
  最初の躓きが大きく減る。

## 2. 採用 / 除外 / 追加調査の判断リスト

改善は「**評価で見つかった、明示不足を埋めるドキュメント修正**」に絞る。理由:
実害の大きい順が(a)既定 backend の前提未記載=必ず躓く >(b)データ送信の未告知=
プライバシー >(c)スコープ未明示=誤解、であり、いずれも**コード挙動は正しく、
文書の欠落が問題**。低リスク・高価値・relay で検証可能。

### 採用(今回実装)

- **D1 [採用]** 既定 `claude-cli` backend の前提を明記。
  - README §セットアップ・getting-started §1 に「既定 backend は Claude Code(`claude` CLI)の
    インストール+認証が前提。不要にするには `.env`+`config.yaml llm.backend: api`」を追記。
  - getting-started §8 トラブルシュートに「`claude が見つかりません`」項を追加。
  - 理由: 一般ユーザーが**最初に必ず踏む段差**。1行の前提明記で回避できる。
- **D2 [採用]** データフロー / プライバシー節を新設。
  - 「ローカルに留まるもの(映像・画面・音声・whisper 文字起こし)」と「外部(Anthropic)へ
    出るもの(plan/remeta/plan-shorts が送る transcript テキスト+`brief.md`)」を対比。
    api backend 時の従量課金・キー管理も一言。
  - 理由: 良いプライバシー設計を**正しく伝える**ことが、機密収録のある一般ユーザーの信頼に直結。
- **D3 [採用]** 「スコープ / 汎用NLEとの違い」節を新設。
  - 「できること一覧」と「意図的に持たない機能(速度変更・カラグレ・トランジション集・
    マスク/クロマキー・音声エフェクト等)と代替・据え置き根拠」を明記。
  - 理由: #5 の誤解を防ぎ、期待値を正しく設定する。next-features-design.md への導線も張る。

### 追加調査 / 据え置き(記録のみ、今回は実装しない)

## 3. フェーズ分けされた実装計画(小さい単位・1タスク=1コミット)

- **各ゲートの検証(コーディネータ)**: `npm test` と `tsc --noEmit` が緑のまま /
