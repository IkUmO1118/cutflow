基盤はかなり揃った。次の課題は、AI編集をどう製品化するか。

この版は 2026-07-06 時点の診断を現行実装へ反映したものです。7/6 版で欠落としていた approved のコード強制、stable id / @id、apply、MCP、OCR、語単位 timestamp、rules / learn、annotation、blur / mosaic、caption anim / karaoke、frames server、素材知覚は既に実装済みです。

総合判定

「Composer / Agent 以前」ではもうない。現在の cutflow は Agent-ready NLE と呼べる段階にある。

最大の未達

不足しているのは基盤ではなく GUI 内 AI 指示、diff 承認フローの主経路化、対話一発編集の製品化 である。

判断

旧ロードマップを盲目的に進めるのは不適切。終わった基盤工事より、UX と製品フローへの変換を優先すべきだ。

Core Finding

ボトルネックは「安全に編集できるか」から「AI 編集をどう製品化するか」へ移った

7/6 版の主要批判だった「plan が盲目で聾」「承認が紳士協定」「行動層が生 JSON 手書き」は、現状にはそのまま当てはまらない。

  • 承認ゲートは approvals.json のハッシュ一致で強制され、boolean approved にはフォールバックしない。
  • apply は validate 前置のアトミック適用で、approved 変更を拒否する。
  • MCP server により、読取系と安全編集系の tool が露出している。
  • stable id と @id によるアドレッシングがある。
  • plan には opt-in で audio / system speech / OCR が接続されている。
  • GUI には three-way diff と conflict resolution がある。

現在の問題は別の場所にある。AI を GUI の中でどう働かせるか、編集結果をどう自己検証させるか、この基盤をどう「1本仕上げる製品体験」に束ねるか、である。

A. AI の知覚強化

B. AI の行動インターフェース

完了 MCP、apply、approved 書換拒否、stable ID / @id、read 系 tool 群。

残課題 道具は安全だが低水準で、task-level tool はまだ薄い。GUI から直接使えない。

C. 編集表現力

完了 annotation、blur / mosaic、caption anim / karaoke、color filter。

残課題 速度変更、フリーズフレーム、汎用キーフレーム基盤、マスク、トラッキング、オートリフレームが無い。時間変化の基盤がボトルネック。

D. 高速フィードバックループ

完了 frames-serve、assert、av probe、stale 検知強化。

残課題 AI が見るのは still と probe であり、動画再生ではない。音の主観評価は人間終端。

E. ローカル×持ち込み AI の運用基盤

完了 AGENTS_CONTRACT.md、JSON Schema 群、rules.md、learn。

残課題 rules は提案生成までで、自動反映や継続メモリはない。収録横断 retrieval もない。

F. ポジショニング / 差別化

強みは変わらない。video-as-code、local-first、approval boundary、agent-editable の組み合わせは依然強い。

残課題 競合の table stakes になった「言えばやる」体験が未製品化。

Priority

盲目的に進めず、優先順位を組み替えるべき理由

  1. 7/6 の欠落一覧を前提にすると、MCP、apply、approval gate、OCR、rules、annotation など、既に終わった領域へ工数を入れ続けてしまう。
  2. 現在のボトルネックは「AI に何ができるか」ではなく、「ユーザーがどう AI を使って 1 本を仕上げるか」である。
  3. 速度変更とキーフレームは周辺改善の延長ではなく、独立した重い設計課題である。

P0

  1. GUI 内 AI 指示チャネル
  2. AI 編集の常設 diff review フロー
  3. plan 知覚経路の既定オン化または明示
  4. 対話一発編集の workflow 化

P1

  1. 高水準 editing tools
  2. 実 A/V フィードバック
  3. material / recording 横断 retrieval

P2

  1. キーフレーム基盤
  2. 速度変更
  3. BYO-AI 拡張

Position

今の cutflow をどう呼ぶべきか

現在の cutflow は、もう「盲目な一発生成ボタン付きリポジトリ」ではない。安全な編集 API があり、承認境界がコードで強制され、agent-friendly な addressing と MCP があり、目と耳も低帯域ながら接続され始めている。

したがって、いま名乗るべきは「動画編集の Cursor の土台」ではなく、Agent-ready NLE である。

次に勝負すべきは基盤追加ではない。GUI 内 AI 指示、AI 編集の差分承認フロー、実 A/V の再観測ループ、対話一発編集の製品化、そしてキーフレーム / 速度変更という表現力の本丸である。
