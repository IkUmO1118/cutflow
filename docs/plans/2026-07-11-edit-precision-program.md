# 編集精度プログラム — カット / 素材 / BGM

> 状態: **稼働中の作業母艦**(初版 2026-07-11)。単発の feature 設計ではなく、
> 「AI 編集の精度を上げる」取り組みを継続的にブラッシュアップするための生きた
> ドキュメント。施策は §4 のバックログで状態管理し、意思決定は §6 に追記する。
>
> **前提となる方針**(ユーザー確認済み・2026-07-11): 「編集できる動画をリッチに
> する(速度/キーフレーム/トランジション等の表現力)」より **「AI の知覚を強化
> して AI 編集の精度を上げる」を優先する**。現状の自動編集が雑(テキトウ)である
> ことが出発点。

---

## 1. 目的とスコープ

**目的**: カット・素材・BGM の3軸で、AI が下す編集判断の精度を上げる。手段は
表現力の追加ではなく **知覚(perception)の強化**。

**スコープ**:

- **カット** — どこを残し・どこを切るか(`cutplan.json`)。現在 AI が決めるが知覚が薄い
- **素材(B-roll / インサート)** — どこに何の素材を挿すか(`overlays.json`)。**現在 AI は不在**
- **BGM** — どの区間にどんな曲を敷くか(`bgm.json`)。**現在 AI は不在**

**スコープ外**(別ロードマップ): 速度変更・キーフレーム波及・トランジション・
マスク等の表現力(`docs/reviews/2026-07-06-ai-native-nle-diagnosis.md` および
`docs/next-features-design.md` 参照)。

---

## 2. 共通原則(3軸すべてに効く設計指針)

精度が落ちる箇所は毎回この4層のどこかに落ちる。施策はこの層に紐づけて考える。

1. **候補集合を意味で作る(garbage-in を消す)**
   AI が選ぶ選択肢そのものが貧しいと、下流がいくら賢くても精度は上がらない。
   候補は音響ではなく **意味(発話内容・語タイムスタンプ・画面・素材メタ)** に
   沿って生成する。

2. **判断入力を高帯域にする(低帯域テキストから脱却)**
   スカラ数値とテキストだけでなく、**実画像・話速・音量・重複・素材の実尺** を
   判断 LLM に渡す。`ai.routes.vision` 基盤は既にあるのに cut 判断で未使用。

3. **決定後に自己検証してループを回す**
   出力を review pipeline(deterministic + VLM)へ回し、悪い判断を機械検出して
   再生成する。`planLoop` / VLM 二次観測 / `review` の基盤はあるが cut の主経路に
   組まれていない。

4. **ハルシネーション耐性を壊さない(番号選択方式の維持)**
   `docs/decisions.md`(2026-07-02)の設計上、LLM には時刻を生成させず候補番号を
   選ばせる(存在しない番号=機械検出可能)。**知覚強化の方向は「LLM に自由に
   書かせる」ではなく「候補集合を意味的に豊かにする(良い番号を増やす)」**。
   素材・BGM に AI を広げるときも同じ原則を踏襲する(AI は列挙された候補から
   選ぶ / 既存アンカーを指す)。

### 3.4 「AI 編集の精度が悪い」と感じる原因の深掘り

結論: 低精度感の主因は単一のモデル性能ではなく、**候補粒度・入力帯域・検証粒度・
ユーザー期待のズレが重なっていること**。現状の JSON は壊れにくいが、壊れていない
ことと「編集として自然である」ことの間にまだ大きな隙間がある。

| 低精度に見える体感 | 実装上の起点 | なぜ悪く見えるか |
|---|---|---|
| 話の途中で切れた / 切り残しがある | 候補は `detect` の無音補集合。境界は `padSec` 固定で、語境界へ snap しない | 無音境界は「話の意味の境界」と一致しない。固定余白は安全だが、語尾・呼吸・次文頭を半端に巻き込む |
| 短い一言・リアクション・クリック直後の声が消える | `detect.complement` は `minKeepSec` 未満の keep 断片を捨てる | 呼吸音対策としては正しいが、短い「できた」「ここです」「あっ」も候補集合から消える。候補に無いものは LLM が復活できない |
| AI が見ている発話と実際の残り方がずれる | `numberSegments` は候補区間に overlap した transcript segment の **全文** を渡す | 候補が文の一部だけを含む場合でも LLM には全文が見える。結果として「残したつもりの説明」が実際は途中から/途中までになる |
| 同じ説明や言い直しが残る | near-duplicate span の検出・グルーピングが無い | プロンプトは「後の綺麗な方を残す」と言うが、LLM は長い候補リストから自力で重複ペアを探す必要がある |
| 重要なデモ・エラー・画面変化を外す | plan の画像入力は未使用。OCR は代表フレーム中点1枚・テキスト化のみ | 開発動画の見せ場は画面変化・エラー表示・実行結果に出ることが多い。中点 OCR だけだと開始/終了付近や動きの意味を落とす |
| 短い見せ場より長い退屈区間を見てしまう | OCR 対象は `ocrMaxSegments` 件までで、超過時は尺の長い区間を優先する | 長い区間は情報量が多いとは限らない。短いエラー表示・成功表示・UI 変化が OCR 対象から落ちると、AI は「画面を見た」つもりで見せ場を外す |
| 環境や設定で精度が急に落ちる | 古い config では `plan.perception` 自体が無いと全オフ。標準でも `systemSpeech` は false。OCR は macOS/Vision 依存で失敗時に劣化する | 同じモデルでも、audio/OCR/system 音声の有無で判断材料が変わる。ユーザーには「AI の精度が不安定」に見える |
| テンポが悪い / 間が詰まらない | plan perception の音は `len/gapBefore/silenceWithin` のスカラ中心。`av.probe` は主経路外 | 「どこに何秒の間があるか」「待ち画面が何秒止まるか」「音が被るか」は観測可能だが、初回判断の標準ループには入っていない |
| 「ここに注釈/素材」を外す | GUI AI の global context は要約 + 上限付き timeline candidates。実フレームは review/確認側 | タイミング候補は選べても、座標や視覚的な邪魔さは推測になる。VLM は明示 action まで走らない |
| グローバル指示の対象時刻が外れる | GUI AI の global timeline candidates は visible captions / keeps / chapters を先頭から上限件数だけ渡す | 「一番良いタイミングに注釈」「目立つ場面に素材」のような全体探索では、後半の候補がそもそも文脈に入らないことがある |
| ズーム・ぼかし・注釈の対象やタイミングがズレる | `describe --json` は `zooms/blurs/annotations` の時刻・矩形を出せるが、提案時の主入力は構造情報中心。blur/annotation は task-level intent がある一方、zoom は patch 寄り | 画面上の「どこを寄る/隠す/指すか」は実ピクセル依存。OCR と矩形だけでは、重要 UI・字幕・既存演出との重なりや、ズーム中の固定px演出のズレを判断しにくい |
| 座標は valid だが対象を外している | GUI AI は出力解像度内へ座標を clamp し、validate も構造中心に検査する | 画面外事故は減るが、clamp 後に矢印・blur・zoom が「別の場所を指す/隠す/寄る」状態でも JSON としては通りうる |
| 演出トラックが過剰 / 足りない | 「この場面は zoom / blur / arrow / spotlight / 何もしない」の候補集合を作る主体が無い | AI はユーザー指示に反応して個別演出を足せるが、動画全体の見せ場・秘匿箇所・注視誘導をスキャンして演出トラックを設計していない |
| 素材・BGM が合っていない | 素材配置と BGM 配置は AI 主経路が無い | ユーザーは「AI 動画編集」として全体を評価するが、現状 AI が責任を持つのは主に cut/meta。未自動化領域も精度不足として体感される |
| ショートの見せ場選定が弱い | `plan-shorts` は `numberSegments` を流用し、現状 perception を渡さない。選択区間は時系列固定で、尺超過時は末尾を落とす | 本編よりさらに画面・音・システム発話を見ない。フックを後半から先頭へ持ってくる編集もできないため、「AI が見せ場を分かっていない」に見える |
| cut と章・タイトル生成が互いに薄まる | 通常 `plan` は cut / chapters / titles / description を1つのプロンプトで作る。`plan --cuts-only` はあるが別経路 | LLM の出力は valid でも、カットの精査とメタ生成が同時タスクになる。章テロップ生成の副作用も含め、ユーザーには編集判断の焦点が甘く見える |
| JSON は valid なのに編集が変 | `validate`/`assert` は構造・存在・尺・一部視覚を検査。意味連続性は薄い | 参照語だけ残る、結論だけ消える、文脈が飛ぶ、素材が話題と合わない等は schema 的には壊れていない |
| なぜ残したのか / 迷ったのか分からない | `cutplan.json` の reason は cut 側だけ。keep の根拠・不確実性・代替案は残らない | 失敗調査が「切った理由」中心になる。残しすぎ・テンポ悪化・迷いどころの原因が plan.raw.txt 依存になり、AI が雑に見える |
| レビューしても信頼しにくい | AI review は改善中だが、変更理由と映像差分の対応がまだ主役ではない | 人間が「この変更は意図通り」と映像で即確認できないと、正しい提案でも精度が低く見える |

#### 3.4.1 上流で起きている5つのズレ

1. **候補集合の recall 不足**
   無音検出は安定しているが、意味的な候補を十分に拾わない。良い境界・良い重複ペア・
   良い素材/BGM/演出トラックのアンカーが候補に無ければ、番号選択方式の LLM は
   正解を選べない。`minKeepSec` で落ちた短い断片も、ここで不可逆に失われる。

2. **候補の意味表示と実映像のズレ**
   `#id [start-end] text` は人間にも AI にも読みやすいが、text は区間内で実際に残る
   語だけに切られていない。ここは低帯域というより **誤った高信号** になりうる。

3. **判断材料の帯域不足**
   transcript/OCR/audio scalar は有用だが、実画像・複数時点の画面変化・話速・音量・
   system 音・freeze・BGM duck・演出トラック同士の重なりまでは標準の判断に
   入っていない。特に開発系動画は「発話していないが画面で重要なことが起きる」
   時間が多く、zoom/blur/annotation の良し悪しも実画面依存になりやすい。
   さらに OCR は対象数・行数・環境で欠落し、GUI の global context も上限付きなので、
   「全体を見た判断」にはなっていない場合がある。

4. **検証が編集の意味まで届いていない**
   現在の検証は「壊れない」「レンダーできる」「指定した構造意図を満たす」に強い。
   しかし「会話として自然」「参照語が宙に浮かない」「デモ結果と説明がセットで残る」
   「素材/BGM が話題に合う」「注釈が邪魔をしていない」「ぼかしが秘匿箇所を覆っている」
   「ズームと固定px演出が矛盾していない」は、まだ assertion / review の一次チェックではない。

5. **検品情報が判断へ戻りきっていない**
   `reviewEvents` には zoom/blur/annotation/cut/BGM の検品観点があるが、主に人間レビューの
   表示用で、提案前の候補生成や plan の再調整に必ず戻るわけではない。VLM secondary も
   「観測」であり、パッチの正規ソースではないため、良い気づきがあっても自動で精度改善へ
   閉じないケースが残る。

#### 3.4.2a 最深の根因(コード実測で新規判明・2026-07-11 追調査)

§3.4.1 の各ズレの**下**に、より構造的な2つの上限がある。これらはモデル性能でも
知覚帯域でもなく、**パイプラインの形そのものが精度の天井を決めている**ため、
他の施策より上流にある。

**R0. カット解像度 = 候補格子の解像度(非分割の壁)**
`buildCutplan`(`plan.ts:141`)は `numbered`(候補)を **1対1** で keep/cut に
写像する。LLM 出力は `cuts: [{id, reason}]` だけで、**候補の内部に新しい境界を
作る手段が無い**。つまり:
- カットの最小単位 = `detect` が作る**無音境界の候補**そのもの。候補境界は
  `minSilenceSec = 0.7`(`config.yaml:43`)以上の無音でしか生まれない。
  **0.7 秒未満のポーズしか挟まない文境界・言い直し境界はそもそも候補に
  存在せず、LLM がどれだけ賢くても切れない。**
- 40 秒ほぼ連続で喋る中央に1文だけ冗長がある場合、その1文だけを消すのは
  **構造的に不可能**(候補が丸ごと1つで all-or-nothing)。速く/詰めて喋る
  場面ほど候補が粗くなり、カットは「大きな塊を残すか消すか」に退化する。
- `plan --cuts-only` の critique ループ(`renderCritiquePrompt`)も**同じ候補
  格子の中で再選択するだけ**で新境界を作れない。自己修正が解像度を上げない。

→ 精度の天井を上げる唯一の道は **候補格子を細かくすること**(C1/C7/C8=語
タイムスタンプで文・節・言い直し境界を候補に足す)。C3/C4/C6/C9/C11 等は
「格子内でより良い番号を選ぶ」に留まる。**C1 系が横断施策より上流にある**
ことがこの調査で確定した。

**R1. C1 の前提資産 `transcript.words` は既定では生成すらされない**
§3.1 は `transcript.words` を「遊休資産」と書いたが実測はより厳しい:
`config.yaml:34 whisper.wordTimestamps: false` が既定。words は
`wordTimestamps: true` にして whisper を `-oj → -ojf` に切り替え**再 transcribe**
したときだけ付く(`transcribe.ts:98,118`)。よって C1/C7/C8 は「配線するだけ」
ではなく **「まず語タイムスタンプを既定で持つ状態にする」前提(下記 W0)が要る**。
明記しないと words の無い収録で静かに劣化し効果測定が濁る。

#### 3.4.2b 既定値とプロンプトが「緩い側」に押している

R0 の粗い格子の中で、既定設定はさらに**カットしない側**へ寄せている:
- `minSilenceSec 0.7` は文境界を取りこぼす(recall 不足)。`padSec 0.15` 固定
  (`config.yaml:45`)= 語/呼吸境界に無関係な一律余白。
- プロンプト `plan-cuts.md:25`「**迷ったら残す。過剰カットより冗長の方がまし**」。
  安全側として正しいが、粗い格子と合わさると実質「ほぼ何も切らない」に倒れやすい。
- `targetOutDurationSec` は既定コメントアウト(`config.yaml:192`)=ペース目標も
  累積尺の圧力も既定では LLM に渡らない。
→ 「粗い格子」×「緩いプロンプト」×「目標尺なし」で**既定の自動カットは構造的に
消極的**。「テンポ良く」を期待する場面で「テキトウ=何もしていない」に見える体感は
ここから来る(X4 の保守度切替が効く層)。

#### 3.4.2c 候補テキストが whisper チャンク粒度で二重に見える

`numberSegments`(`plan.ts:85`)の overlap 条件 `s.start < k.end && s.end > k.start`
は、候補に**少しでも重なる** whisper segment の**全文**を `.join(" ")` で入れる。
whisper segment は 5〜15 秒チャンクなので、境界をまたぐチャンクは**両隣の候補に
全文重複**して現れる。LLM は「実際には残らない語」を含む**架空の transcript** を
根拠に判断する(C8 が指す歪み。R0 と合流すると、候補が粗い上に表示テキストも
境界で嘘をつく二重苦)。

#### 3.4.2d 知覚が長い候補・中点に偏る

- `selectOcrTargets`(`perception.ts:129`)は上限超のとき**尺の長い順**に OCR
  対象を選ぶ。**短い候補(反応・エラー表示・一瞬のデモ結果)は視覚知覚ゼロ**に
  なりうる。開発動画の見せ場は短い瞬間に出やすく取りこぼしと逆相関。
- OCR は**中点1枚**(`representativeSourceTime`)。長い候補では任意の1点で、
  開始/終了の画面変化を落とす(C9 が指す)。
- `reason` は自由文で**未検証**。「言い直しの前半」と書いても near-dup かを検査
  する主体はいない(X2/X3=証拠添付と assert 化の根拠)。

#### 3.4.2 実装から見える重要な示唆

- **`numberSegments` の全文 overlap 問題は優先度が高い**。これは LLM 能力以前に入力表現の
  歪みで、境界精密化(C7)や語候補化(C1)と同時に直す価値がある。
- **「迷ったら残す」は安全だが、低精度感を増やす**。過剰カット事故を防ぐ方針として
  正しい一方、ユーザーが「テンポ良くして」と期待している場面では「何もしていない」
  印象になる。rules/brief/目標尺で保守度を切り替えられる必要がある。
- **観測コマンドと判断ループの距離がある**。`materials` / `av` / `frames` / `assert`
  は既に強いが、明示的に叩いた時だけ効く。精度を上げるには「読める」から
  「判断前後で必ず読む」へ進める必要がある。
- **観測カバレッジを可視化しないと、モデル問題と入力欠落を誤認する**。OCR が何区間に
  走ったか、何区間が未観測か、systemSpeech が無いのか off なのか、global context が
  何件で切れたのかを plan/review に残さないと、低精度の原因調査が毎回曖昧になる。
- **素材/BGM は「精度が悪い」のではなく「AI 所掌外」**。ただしユーザーの体感では
  所掌外も AI 編集品質に含まれるため、§4 の素材/BGM バックログを精度施策として扱う。
- **演出トラックは「何をするか」と「どこへ置くか」を分ける必要がある**。zoom は
  詳細を見せる、blur は隠す、annotation は注視させる、という目的が違う。現状は
  ユーザー指示に対する個別 patch/intent になりやすく、動画全体から「この場面には
  どの演出が必要か」を候補化する層が薄い。
- **zoom/blur/annotation は座標だけ valid でも品質が出ない**。出力px内に収まることは
  必要条件でしかなく、字幕・重要 UI・既存素材・ズーム区間との相互作用を見ないと
  「邪魔」「隠れていない」「指している場所が違う」になる。
- **ショートは本編カットの弱点を増幅する**。同じ `numberSegments` 表現を使いながら
  perception が無く、さらに時系列固定・末尾削除の制約があるため、見せ場選定の失敗が
  より「AI が内容を理解していない」印象になりやすい。
- **keep 側の説明責任が不足している**。切った理由だけでなく「残した理由」「迷った区間」
  「判断材料が不足した区間」を残すと、レビューと次施策の因果が追いやすくなる。
- **レビュー UX も精度の一部**。JSON diff 中心だと、人間は採用判断に自信を持てない。
  AI が正しくても「検品できない」状態は低精度として認識される。

## 3.5 ハーネス設計の観点(cut 判断は現状エージェントではない)

Claude Code 流出が示した教訓は「**ループ + tool + 検証** が、より大きな単発
プロンプトに勝つ」。cutflow の cut 判断は**その負ける側=単発プロンプト方式**である
ことがコード実測で確定した(2026-07-11 追調査)。

**実測した現状のハーネス:**
- `plan` は `completeWithJsonSchema`(`plan.ts:359`)→ アダプタが `claude -p
  --output-format text`(`registry.ts:60`)または `/v1/messages` を**1回**叩くだけ。
  **tool なし・ループなし・単発 structured completion**。
- 判断 LLM が受け取るのは `perception.ts` が**事前計算して文字列に焼き込んだ知覚**
  だけ(push 型・一律・OCR 長尺偏り・中点1枚)。**判断者は自分で `frames`/`describe`/
  `av`/`materials`/`ocr` を引けない**。
- opt-in の critique ループ(`runCutsLoop`)も**焼き込みプロンプトを作り直すだけ**で
  tool は握らせない。
- 強い知覚・編集 tool 群(`frames`/`describe`/`av`/`materials`/`assert`/`apply`)は
  既にあるが、握っているのは**人間か外部 MCP エージェント**であって cut 判断 LLM ではない。

**ハーネスが効く軸は R0 と直交する(切り分けが要):**

| 軸 | 中身 | ハーネスで上がるか |
|---|---|---|
| A. 行動空間の解像度(R0) | LLM が「何を表現できるか」。今は候補を丸ごと keep/cut | **上がらない**。出力が「候補id選択」なら何ターン回しても無音格子で頭打ち。→ C1/W0 の領分 |
| B. 行動空間の中での判断品質 | 与えられた選択肢で良い選択をするか | **大きく上がる。ほぼ手つかず** |

**最深の含意 — agentic ハーネスは既に別経路として存在する:**
cutflow には2つの AI 経路がある。**(a) 内部 `plan` LLM**(非エージェント・番号選択・
R0 の原因・安全だが解像度が天井)と、**(b) 外部 MCP エージェント**(`describe`/`frames`/
`apply` を tool として叩ける本物のハーネス。`apply` は**候補格子に縛られず任意区間を
書け**、書込前に validate/assert が効く=検査が安全網)。**(b) は原理的に R0 を回避できる**
(候補内部を割った segment を `apply` で書けるため)。番号選択方式は「時刻をでっち上げ
させない」ための**ハーネス設計上の選択**であり、`apply`+検査という別の安全網に替えれば
解像度を犠牲にせずハルシネーションを塞げる。**部品は全部揃っていて、まだ『tool を使う
検証ループとして cut 選択を駆動する』形に組み上げていないだけ**。

**限界(正直に):**
- ハーネスは W0/C1 の代替ではない。格子が粗ければ完璧なエージェントでも文中を切れない
  = **補完関係**(C1 が行動空間を広げ、ハーネスが使いこなす)。ただし (b) 経路を選べば
  ハーネス側が R0 を直接崩す。
- コスト: 候補ごとに headless-chrome で frames を撮るループはローカルでは遅い。plan 1回の
  token/レイテンシが跳ねる。
- 再現性: 単発+opt-in知覚は今バイト等価を保証。agentic を既定にすると再現性が崩れるので
  `plan.perception` と同じく opt-in/sticky を守る。

## 4. 施策バックログ(効き順・状態管理)

状態: `todo` / `in-progress` / `done` / `parked`。効きは編集精度への寄与、
コストは実装規模。**上ほど効いて安い**順に並べる。ブラッシュアップのたびにここを更新する。

> **確定方針(2026-07-11・ユーザー判断。詳細は §6):**
> - **アーキ = ハイブリッド(D1)**: 番号選択方式を安全網として保ちつつ、**確信の高い
>   区間だけ `apply`+validate/assert で候補内部分割**を許す。R0 を安全に部分突破する
>   (H6 がこの経路。番号選択の完全放棄=apply全面移行は採らない)。

>   脱線は切り見せ場は残す。rules/brief/`targetOutDurationSec` で safe↔aggressive 上書き(X4)。
> - **スコープ = cut に集中(D5)**: 素材(M)/BGM(B)/演出(E)は cut の基盤が固まるまで parked 寄り。
> - **順序(D6) = W0 → C1(格子拡張)を先行、その後 H1/H2(tool+検証ループ)を必ず行う。**
>   格子で行動空間を広げてから、ハーネスでそれを使いこなす。

## 5. 精度の測り方(ブラッシュアップには測定が要る)

施策の効果は主観だけで判断しない。以下を基準線として整備する。

- **決定論アサーション**(`assert` / `assertions.json`): 「語の途中で切れていない」
  「keep 区間に実発話がある」「素材の尺超過がない」「BGM が覆っていない区間が
  意図的」等を機械検証。C6/M2/B2 の受け入れ基準に使う。
- **精度エラー分類**: 人間が直した箇所を `boundary`(境界) / `duplicate`(重複)
  / `visual-miss`(画面見落とし) / `context-break`(文脈破綻) /
  `effect-mismatch`(zoom/blur/annotation の種別・時刻・座標の不一致) /
  `material-mismatch` / `bgm-mismatch` / `review-unclear` に分類する。施策が
  どの体感問題を減らしたかを §6 に残す。
- **境界評価**: cut 境界ごとに「語途中」「前後の指示語/未完了文」「直前直後の
  画面/音の飛び」を記録する。`numberSegments` の全文 overlap 問題(C8)と
  境界 snap(C7)の効果測定に使う。
- **review bundle**(`review` コマンド + VLM 二次観測): before/after の deterministic
  観測 + 視覚チェック。施策前後の比較に使う。
- **観測カバレッジ**: plan/review 実行ごとに、候補数、`minKeepSec` で捨てた断片数、
  OCR 実行区間数/未実行区間数、OCR 失敗/非対応、systemSpeech の有無、VLM 実行有無、
  global timeline candidates の打ち切り有無を記録する。低精度の原因を「判断ミス」と
  「入力に無かった」に分ける。
- **回帰サンプル**: 代表収録を数本固定し、施策ごとに `describe --json` の diff と
  人間評価(1〜5)を §6 に記録する。中立 cwd から LLM を走らせる
  (`memory/llm-command-verify-neutral-cwd.md` 参照)。
- **検品しやすさ**: AI 提案が正しいかだけでなく、人間が採用判断できるかを測る。
  例: 推奨確認時刻が十分か、before/after が意図に対応しているか、JSON diff を読まずに
  採否できたか。
- **演出トラック評価**: zoom/blur/annotation について、種別選択(必要/不要/別種が適切)、
  時刻、座標、他トラックとの重なり、zoom 中の追従ズレを別々に採点する。座標が valid
  でも「見せたい/隠したい/指したい」目的を満たさなければ fail とする。
- **keep 理由・不確実性**: cut された区間だけでなく、keep された長尺区間・迷った区間・
  観測不足区間をレビュー対象として数える。残しすぎによる低精度感を測れるようにする。
- **ショート評価**: plan-shorts について、フックの強さ、結論/解決の保持、60秒調整で
  落ちた区間、画面見せ場の保持、本編と同じ秘匿/注釈/BGM 要件の継承漏れを別採点する。
- **バイト等価の担保**: 知覚を足すときは opt-in / sticky を守り、無効時に既存挙動と
  バイト等価であることを確認する(`plan.perception` の設計に倣う)。
