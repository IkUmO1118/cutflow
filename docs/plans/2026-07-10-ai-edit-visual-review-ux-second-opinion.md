# AI 編集レビュー画面の UX 改善検討メモ

*2026-07-10 / セカンドオピニオン収集用*

対象:

- GUI の「AI 編集後の diff / review 画面」
- AI が提案した JSON 変更を、人間が採用・却下・再調整する確認体験
- 特に overlays / annotations / chapters / transcript segments など、動画上の見た目に影響する変更

関連設計:

- `docs/plans/2026-07-07-editor-diff-review-design.md`
- `docs/plans/2026-07-08-editor-ai-command-proposal-review-design.md`
- `docs/plans/2026-07-09-p2-vlm-secondary-observation-agent-gui-design.md`

---

## 1. 背景

現行の AI 編集レビュー画面は、AI が生成した変更を「JSON diff に近い文字列」として
左右に表示している。技術的には何が変わったかを確認できるが、動画編集の確認画面としては
認知負荷が高い。

ユーザーが本当に判断したいのは、次の3点である。

- 映像がどう変わったか
- その変更が意図に合っているか
- 採用してよいか、再調整すべきか、却下すべきか

一方、現在の画面は「どの JSON がどう変わったか」を中心に構成されている。そのため、
動画編集者や非実装者にとっては、採用判断に必要な情報へ到達しづらい。

---

## 2. 現状画面の主な課題

### 2.1 文字情報が多すぎる

上部に検査警告、AI メモ、確認推奨時刻、チェックボックス、操作ボタンが並び、その下に
JSON の左右 diff が表示される。重要情報が多いこと自体は正しいが、初見で何を見るべきかが
分かりにくい。

特に問題になっている点:

- JSON が主役になっており、映像の変化が主役になっていない
- 警告、説明、diff、操作ボタンが同じ画面密度で並んでいる
- 「確認推奨」の時刻がテキストとして表示されるだけで、動画プレビューやタイムラインと接続していない
- 変更理由と実際の見た目の対応が分かりにくい

### 2.2 左右 diff が動画編集の判断に向いていない

現在の左右比較は、Cursor などのコード diff に近い構造である。コード編集では有効だが、
動画編集では JSON の値だけを見ても採用判断が難しい。

例:

- `annotations` に arrow が追加された
- `chapters` のタイトルや時刻が変わった
- `transcript segments` の文言や範囲が変わった

これらは JSON として読むより、動画上で「どこに、いつ、どう見えるか」を見た方が早い。

### 2.3 ボタンの意味が近すぎる

現在は次のような操作が並ぶ。

- 比較を生成
- 適用のみ
- 適用して保存
- 適用して確認

それぞれのシステム上の意味は違うが、ユーザーの意思決定としては近すぎる。

ユーザー視点では、まず必要なのは次の判断である。

- この変更を採用する
- この変更を却下する
- AI に再調整させる
- 詳細を確認する

「適用」と「保存」を強く分ける UI は、内部状態をユーザーに露出しすぎている可能性がある。

### 2.4 チェックボックスの導線が曖昧

「画像も AI に確認させる」のようなチェックボックスは、機能としては重要だが、
レビュー画面の主導線に置くと迷いを生む。

問題:

- このチェックは採用前に必ず必要なのかが分かりにくい
- コスト、外部送信、処理時間の意味が重いのに、UI 上は軽く見える
- 変更を採用するための操作なのか、再調整のための操作なのかが曖昧

---

## 3. セカンドオピニオン結論

現行メモの方向性は正しい。ただし「左右 diff 画面を改善する」という捉え方だと、
コード diff の比喩を引きずりやすい。より良い目標は、**AI 編集の検品モード**として
再設計することである。

この画面でユーザーがしたいことは、JSON の差分確認ではなく次の判断である。

- AI 編集後の映像が意図に合っているか見る
- 気になる変更だけ before / after で確認する
- 変更ごとに使う / 使わない / AI に直させるを決める
- 最後に一度だけ内容を確定する

したがって、主役は「左右 diff」ではなく **AI 編集後の大きなプレビュー** とする。
before / after は常時2画面で見せるのではなく、必要なときに切り替える比較モードにする。

基本方針:

- 主役は AI 編集後プレビュー
- before / after は比較モードとして提供する
- 差分は JSON hunk ではなく編集イベントとして見せる
- ユーザーの操作は `使う` / `使わない` / `AI に直させる` / `確定` に寄せる
- JSON diff は詳細情報として折りたたむ
- AI / VLM / 検査警告は採用判断の補助情報として扱う

## 4. 採用する画面構成

### 4.1 最上位の情報設計

推奨するレビュー画面は、3ペインの検品モードである。

```text
AI 編集レビュー

┌──────────────┬──────────────────────────────┬──────────────────┐
│ 変更リスト     │ AI 編集後のプレビュー            │ 選択中の変更       │
│              │                              │                  │
│ 00:08 矢印追加 │ [AI 編集後の映像]                │ 矢印を追加         │
│      要確認    │                              │ 目的: 重要箇所を強調 │
│ 00:34 字幕変更 │ [Beforeを見る] [左右比較] [重ねる] │ 確認: 文字を隠していないか │
│      OK       │                              │                  │
│ 01:13 ぼかし  │ ━━●━━━━━━●━━━━━━━━●━━━━      │ [使わない] [使う]   │
│      要確認    │                              │ [AIに直させる]      │
└──────────────┴──────────────────────────────┴──────────────────┘

[キャンセル]                         [この内容で確定]
[詳細: JSON diff]
```

この構成の狙いは、ユーザーを「差分を読む人」ではなく「映像を検品する人」として扱うこと。
変更リストで全体像をつかみ、中央で映像を見て、右で選択中の変更だけを判断する。

### 4.2 AI 編集後プレビューを主役にする

左右比較を常時主役にすると、画面が狭くなり、初見では何を見ればよいか分かりにくい。
特に素人向けには、まず「AI が作った結果」を大きく見せる方が自然である。

推奨:

- 中央に AI 編集後プレビューを大きく表示する
- 変更マーカーをクリックすると該当時刻へジャンプする
- 追加 / 変更された視覚要素は、可能ならプレビュー上で軽く強調する
- before / after は必要なときに切り替える

比較モード:

- `AI編集後`: 既定。最終的に採用される見た目を大きく見る
- `Beforeを見る`: 押している間、または一時切替で変更前を表示する
- `左右比較`: before / after を並べて見る
- `重ねて比較`: 位置やサイズのズレを確認する

注釈、字幕、ズーム、ぼかしのような視覚差分は、JSON よりも画面上で確認する方が自然である。
ただし比較はあくまで補助であり、最初から左右2画面を強制しない。

### 4.3 変更リストで全体像を見せる

変更が複数ある場合は、1件ずつのウィザードだけでは全体像を失う。一方、一覧だけでは
各変更の検品が浅くなる。推奨は **一覧 + 1件フォーカス** のハイブリッドである。

変更リストに出す情報:

- 時刻
- 変更タイトル
- 種類
- 状態: `未確認` / `OK` / `要確認` / `使わない`
- 警告の有無

例:

```text
00:08.20  矢印を追加       要確認
00:34.60  字幕を短く変更   OK
01:13.40  ぼかしを追加     要確認
```

変更を選ぶと:

- 中央プレビューが該当時刻へシークする
- タイムライン上の該当マーカーを選択する
- 右ペインに目的、確認ポイント、警告、操作を表示する

### 4.4 差分を編集イベントとして見せる

JSON の hunk をそのまま表示するのではなく、ユーザーが理解できる編集単位に変換する。
内部では hunk の集合でも、UI では `ReviewEvent` のような表示用単位に正規化する。

例:

```ts
type ReviewEvent = {
  id: string;
  title: string;          // "矢印を追加"
  kind: "annotation" | "caption" | "cut" | "blur" | "zoom" | "chapter" | "audio";
  timeRange?: { startSec: number; endSec: number };
  hunkLabels: string[];
  intent?: string;
  checkPoints: string[];
  warnings: string[];
  jsonPaths: string[];
};
```

表示例:

```text
矢印を追加
00:08.20 - 00:10.50

目的:
序盤タイトルの導入を強調

確認ポイント:
- 矢印が画面外に出ていないか
- 既存の文字を隠していないか

[使わない] [使う] [AIに直させる]
```

JSON diff は「詳細」に退避する。

```text
[詳細: JSON diff を見る]
```

これにより、通常の採用判断では JSON を読まなくてよくなる。

### 4.5 タイムライン上に変更マーカーを置く

「確認推奨: 00:08.20, 00:34.60, 01:13.40」のようなテキスト表示は、
タイムライン上のマーカーに変換する。

```text
━━●━━━━━━●━━━━━━━━●━━━━
   注釈       字幕        ぼかし
```

マーカーをクリックすると:

- 該当時刻へシーク
- 対応する変更イベントを選択
- before / after 比較の対象時刻を揃える
- その変更に関係する警告だけを表示する

### 4.6 警告は全体要約 + 変更ごとに出す

現在の黄色い検査警告は情報量が多く、画面上部を支配している。警告は重要だが、
最初に見せるべきなのは詳細ログではなく要約である。

例:

```text
要確認 4件
- 画面外の可能性がある注釈 2件
- 字幕と重なっている可能性 1件
- 解決できない確認フレーム 1件

[詳細を見る]
```

詳細ログは折りたたみ、または該当変更を開いたときに表示する。
「どの変更に関係する警告か」が分かることを優先する。

## 5. ボタン設計の見直し

### 5.1 推奨する主操作

主操作は、ユーザーの意思決定に合わせて絞る。

変更単位:

```text
[使わない] [使う] [AIに直させる]
```

レビュー全体:

```text
[キャンセル] [この内容で確定]
```

補助操作:

```text
[詳細を見る]
[JSON diff]
[比較を更新]
```

「採用」「却下」よりも `使う` / `使わない` の方が、動画編集の文脈では分かりやすい。
`AIに直させる` は、単なる検査ではなく再提案の入口として扱う。

### 5.2 「適用」と「保存」の扱い

内部的には「live state へ適用」と「ディスクへ保存」は別である。しかしレビュー UI では、
この違いを前面に出しすぎない方がよい。

推奨:

1. 主ボタンは `この内容で確定` にする
2. 確定時に保存まで行うなら、ボタン付近に「プロジェクトに保存します」と明示する
3. `適用のみ` は通常表示しない。必要なら詳細メニューに退避する
4. 既存の undo / draft / validation gate とは内部で整合させる

避けたい表示:

```text
[適用のみ] [適用して保存] [適用して確認]
```

これらはシステム上の違いとしては正しいが、素人向けの意思決定としては近すぎる。
レビュー画面では「この AI 編集を最終的に使うか」に集中させる。

### 5.3 `比較を生成` の扱い

`比較を生成` はユーザーの目的を表す言葉ではない。可能なら自動生成に寄せる。
手動操作として残す場合も、表現は `比較を更新` 程度にする。

推奨:

- AI 提案が開いたら、必要な still / clip 比較をバックグラウンド生成する
- 生成中は選択中変更に `比較を準備中` と出す
- stale になった場合だけ `比較を更新` を出す
- 比較生成に失敗しても、JSON diff ではなく「この変更の比較を作れませんでした」と表示する

## 6. AI 再チェック / VLM の扱い

「画像も AI に確認させる」は、常時表示のチェックボックスではなく、
`AIに直させる` の中の明示アクションにする。

理由:

- 外部 provider へ画像を送る
- コストや処理時間がある
- 結果は採用判断ではなく補助観測である
- チェックボックスだと、採用前に必須なのか任意なのか分かりにくい

推奨 UI:

```text
AIに直させる

この変更をもう一度見直します。
代表フレーム 3 枚を vision-provider-name に送ります。

[指示を追加]
[画像を見せて再提案]
```

表示すべき情報:

- 何枚の画像を送るか
- どの provider に送るか
- 処理時間の目安
- 結果は自動採用されず、もう一度レビューに戻ること

再調整後も、必ず新しい提案としてレビューに戻す。VLM の観測で自動採用しない。

---

## 7. レビュー単位

現在の diff はファイルや JSON hunk の単位に寄りやすい。UX としては、
次のような編集イベント単位に正規化する。

- 注釈を追加
- 注釈の位置を変更
- 字幕文言を変更
- 字幕の表示範囲を変更
- カット範囲を変更
- ズーム範囲を変更
- ぼかしを追加
- BGM の音量や区間を変更
- チャプター名を変更
- チャプター境界を変更

各イベントに持たせる情報:

- 対象時刻
- 対象要素
- 変更タイトル
- AI の意図
- 確認ポイント
- 検査警告
- 採用状態
- 対応する hunk labels
- 詳細 JSON path

注意:

- `chapters` は現在の `ReviewDocs` / GUI diff 対象には含まれていない。UX で扱うなら、
  データ対象の拡張を別タスクとして設計する。
- 1つの編集イベントが複数 hunk を持ってよい。例: 注釈追加は `start` / `end` / `type` /
  座標 / 色をまとめて1イベントにする。
- hunk 単位の細かい採否は、必要なら詳細モードで提供する。通常ユーザーには出さない。

## 8. 段階的な実装案

### Phase 1: hunk 表示から編集イベント表示へ

まず、既存の JSON hunk UI を主役から外す。

- `ReviewEvent` 生成ロジックを追加する
- hunk を編集イベントにグループ化する
- 変更リストを表示する
- 選択中イベントのタイトル、目的、確認ポイントを表示する
- JSON diff を折りたたむ
- 警告を全体要約 + イベントごとに分ける
- ボタンを `使う / 使わない / この内容で確定` に寄せる

### Phase 2: AI 編集後プレビュー中心の検品画面

既存の still / clip review を、補助情報ではなく選択中変更の visual review として見せる。

- 中央に AI 編集後プレビューを大きく置く
- 変更マーカー付きタイムラインを表示する
- マーカークリックで変更イベントを選択する
- before / after still を選択中イベントに紐づけて表示する
- `Beforeを見る` / `左右比較` / `重ねて比較` の切替 UI を追加する

### Phase 3: 比較生成と stale 状態の整理

比較生成をユーザーの主操作から外す。

- AI 提案後に必要な比較を自動生成する
- 生成中 / 失敗 / stale をイベント単位で表示する
- `比較を生成` ではなく `比較を更新` にする
- 比較が無い状態でも、イベント単位の採否はできるようにする

### Phase 4: AI 再調整導線

視覚確認と再提案をレビュー画面に統合する。

- `AIに直させる` を変更単位の操作として追加する
- 代表フレームを AI に送る前に送信内容を明示する
- 観測結果をユーザーに表示する
- 観測結果を次の AI 提案へ渡す
- 再提案後も必ずレビューに戻す

### Phase 5: 高度な before / after 動画比較

必要になった段階で、より強い比較体験を追加する。

- 左右動画プレビューを同期再生する
- 重ね合わせ比較を追加する
- 変更箇所を視覚的にハイライトする
- 変更単位で before / after clip を検品できるようにする

左右動画比較は重要だが、最初の実装優先度は Phase 1 / 2 より下げる。
先に「JSON を読まなくてよい検品体験」を作る方が効果が大きい。

## 9. セカンドオピニオンの論点への回答

1. レビュー画面の主役は、左右動画比較でよいか
   - 主役は `AI 編集後の大きなプレビュー` がよい。左右比較は必要時の比較モードにする。

2. JSON diff は初期表示から隠してよいか
   - 隠してよい。通常の採用判断では JSON を読ませない。詳細 / デバッグとして残す。

3. `採用` と `保存` を UI 上でどこまで分けるべきか
   - 主導線では分けない方がよい。`この内容で確定` にまとめ、保存する場合は明示する。

4. 変更単位は JSON hunk ではなく編集イベントに変換すべきか
   - 変換すべき。hunk は内部単位、イベントはユーザーの判断単位。

5. 警告は上部に詳細表示するより、変更ごとの要約表示にすべきか
   - 全体要約 + 変更ごとの表示がよい。詳細ログは折りたたむ。

6. VLM / AI 視覚確認はチェックボックスではなく明示アクションにすべきか
   - 明示アクションにすべき。`AIに直させる` の中で、送信内容と provider を表示する。

7. `比較を生成` はユーザー操作として残すべきか、自動生成に寄せるべきか
   - 自動生成に寄せる。手動操作が必要なら `比較を更新` として stale 時に出す。

8. 変更が複数ある場合、1件ずつレビューする方式と一覧方式のどちらがよいか
   - 一覧 + 1件フォーカスのハイブリッドがよい。

9. AI の説明文はどの程度まで表示すべきか
   - `目的` と `確認ポイント` に絞る。長い推論や実装説明は不要。

10. 動画編集者にとって最も安心できる採用フローは何か
    - 変更リストで全体を見て、各変更を映像で確認し、最後に一度だけ確定する流れ。

## 10. 暫定結論

現在の UI は、AI が変更した JSON を確認するための画面としては成立しているが、
AI 編集を人間が検品する画面としては情報設計が逆向きである。

目指すべき方向は次である。

- diff 画面から AI 編集の検品モードへ
- 左右比較常時表示から AI 編集後プレビュー主役へ
- ファイル差分から編集イベントへ
- 内部操作名からユーザーの意思決定へ
- テキスト警告から時刻・画面・タイムライン連動へ
- チェックボックスから明示的な再調整アクションへ

最終的には、ユーザーが JSON を読まずに「映像として良いか」を判断できるレビュー画面にする。
JSON は検証やデバッグのための詳細情報として残すが、通常の採用判断の主役にはしない。
